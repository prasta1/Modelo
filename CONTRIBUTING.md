# Contributing to Modelo

## Workflow

1. **Branch from `main`.** Never commit directly to `main`.
2. **Open a pull request** against `main`. Nothing is merged without a PR.
3. **Keep PRs small.** A PR should be reviewable in full. If your diff is large, split it into focused, incremental PRs.

## Building

Modelo is a native macOS SwiftUI app. The Xcode project is generated by XcodeGen from `project.yml` and is **never committed** (it's gitignored).

```bash
# Prerequisites
brew install xcodegen   # if you don't have it

# Regenerate the project from project.yml
xcodegen generate

# Build (Debug)
xcodebuild -project Modelo.xcodeproj -scheme Modelo -configuration Debug \
  -destination 'platform=macOS' -derivedDataPath build \
  CODE_SIGNING_ALLOWED=NO build
```

## Testing

```bash
xcodebuild -project Modelo.xcodeproj -scheme Modelo \
  -destination 'platform=macOS' -derivedDataPath build \
  CODE_SIGNING_ALLOWED=NO test
```

A green test run is the **minimum bar** for any contribution — if tests fail, your PR is not ready.

## Tests and new features

**If you add or change behavior, you must add (or update) tests.**

- New feature → new test.
- Bug fix → regression test that would have caught it.
- Refactor → existing tests should still pass; add one if the new behavior needs coverage.

Untested changes will be asked to be tested before merging.

## Code style

- Follow existing conventions in the codebase. When in doubt, match the surrounding code.
- Keep diffs focused and readable. One logical change per commit.
- Commit messages should describe **what** and **why**, not just **what changed**.

## Project structure

- **Source:** `Modelo/` — all app code (`Models/`, `Views/`, `Services/`, `Settings/`).
- **Tests:** `ModeloTests/` — test suite, at the repo root alongside `Modelo/`.
- **Manifest:** `project.yml` — XcodeGen project definition. Never edit the generated `.xcodeproj` directly.
- **Dependencies:** Swift Package Manager (MarkdownUI, Highlightr) — add new deps in `project.yml`.

## Submitting a PR

1. Push your branch and open a PR against `main`.
2. Ensure `xcodebuild test` passes locally.
3. Describe the change in the PR body: what it does, why, and how to verify.
4. Link any related issues.

## Getting help

Open an issue or PR with your question. Small, focused questions get faster answers.
